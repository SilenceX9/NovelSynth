# Novel Dehydrator MVP 技术决策

本文档归档 novel-dehydrator-mvp 项目初期拍板的四项技术选型与约定。每项包含：决定内容、拒绝的方案、理由、以及何时重新评估。

---

## 决定 1：后端框架与 LLM 调用方式

### 背景

MVP 需要选择一个后端框架和 LLM 调用方式。本地开发环境为 MacBook Air M4 16G。

### 决定

- 后端使用 **FastAPI**
- LLM 调用走 **直接 API 调用**（openai SDK / requests），不走 Dify 等编排平台

### 理由

- MVP 链路简单：接收文本 → LLM 脱水 → 返回结果，不需要 Dify 的工作流编排能力
- Dify 在本机占用约 1G 内存，对 16G Mac 来说是明显的资源负担
- 直接调 API 延迟更低、调试更直观

### 拒绝的方案

- **Dify 编排**：功能过度、内存占用高，不适合当前简单链路

### 后果与约束

- `llm_client` 层需要保持接口抽象，后续想切换到 Dify 时只需替换该层实现，不影响其他代码

### 何时重新评估

当脱水链路复杂度显著上升（需要多步 LLM 调用、条件分支、人工审核节点等），且团队决定引入 Dify 作为统一编排平台时，重新评估此决定。

## 决定 2：Web UI 技术栈

### 背景

MVP 需要 3 个页面：上传/导入页面、脱水结果展示页面、历史管理页面。

### 决定

使用 **原生 HTML + CSS + JavaScript**，不引入任何构建工具链。页面通过 FastAPI `StaticFiles` 直接挂载。

### 理由

- 3 个页面，交互简单，原生方案零构建、零依赖
- 样式可完全自定义，长文本阅读体验可控
- 开发体验直接：改 HTML 刷新即可，无需等待构建

### 拒绝的方案

- **React + Vite**：引入构建工具链增加维护成本，3 个页面远未到达需要组件化的复杂度阈值
- **Gradio**：样式定制能力弱，长文本阅读场景下体验差

### 何时重新评估

当页面数量超过 10 个、出现跨页面状态管理需求、或需要复杂交互组件（如富文本编辑器、实时协作等）时，重新评估引入前端框架和构建工具链。

## 决定 3：数据存储方案

### 背景

脱水后的文本和相关元数据需要持久化存储和检索。

### 决定

- 元数据存储使用 **SQLite**
- 脱水后的文本按文件结构存储在**本地文件系统**
- 全文检索采用**段落级别 keyword 匹配 + LLM 重排**，不使用向量数据库

### 理由

- MVP 数据量小（单用户级），SQLite 完全胜任
- 向量数据库（Milvus/Chroma）部署维护成本高，在数据量小时优势不明显
- 段落 keyword 匹配 + LLM 重排在准确性上可满足当前需求

### 拒绝的方案

- **Milvus / Chroma 等向量数据库**：部署复杂、资源消耗大，对 MVP 数据量来说过度设计

### 何时重新评估

- 单用户脱水文本总量超过 100 万字，或有多用户并发检索需求时，重新评估向量数据库
- 当 keyword 匹配的检索准确率成为瓶颈时，考虑引入 embedding + 向量检索

## 决定 4：文本切分策略

### 背景

输入的小说文本需要切分为 LLM 可处理的块。

### 决定

- **按章节边界切分**，利用网文天然的章节结构
- 单章超过 **15,000 字**时，按段落进行二次切分
- 不使用固定字数硬切

### 理由

- 网文天然带有章节结构，按章节切分保持语义完整性
- 固定字数硬切会在句子/段落中间断裂，影响 LLM 理解质量
- 15,000 字阈值是当前主流 LLM context window 的安全缓冲区

### 拒绝的方案

- **固定字数切分（如每 4000 字一刀切）**：破坏章节和段落结构，语义碎片化

### 何时重新评估

- 当处理非网文格式文本（无章节结构的散文、论文等）时，需要补充备用切分策略
- 当 LLM context window 显著扩大后，可重新评估切分阈值
